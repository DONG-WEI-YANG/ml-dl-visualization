# 雲端載入韌性與課程入口 UI/UX 設計

日期：2026-07-15

## 背景與問題

正式前端透過 `VITE_API_BASE` 連線至 Hugging Face Space。瀏覽器若保留登入 token，`AuthProvider` 啟動時會呼叫 `/api/auth/me`，而目前請求沒有逾時或取消機制。雲端後端無回應時，`RequireAuth` 會永久顯示 `Loading...`，學生因而無法進入任何課程。

2026-07-15 的實測結果顯示 DNS 約 0.02 秒、TLS 約 0.8 秒，但 `/health` 分別等待 90 秒與 30 秒仍未收到第一個位元組。這表示瓶頸位於雲端後端可服務狀態，而不是前端靜態資源或網路名稱解析。

後端目前在 FastAPI lifespan 內同步執行資料庫初始化、RAG 表格初始化，以及 RAG 為空時的教材讀取與索引。這些工作完成前應用程式不接受請求，是冷啟動期間無法回應的主要程式風險。平台休眠或容器異常仍屬外部風險，因此前端也必須具備容錯能力。

## 目標

- 學生不再因身份驗證請求無回應而永久停留在載入畫面。
- 雲端服務啟動中或故障時，學生仍能進入首頁、18 週課程與不依賴後端的互動視覺化。
- 後端優先提供健康檢查與基本 API，再於背景完成非必要初始化。
- 提供清楚、可操作且符合行動裝置需求的連線狀態。
- 用可觀測資料分辨平台冷啟動、應用程式初始化與一般 API 延遲。

## 非目標

- 本次不更換雲端供應商或購買常駐運算方案。
- 不重寫身份驗證、RAG、測驗或 AI 助教的業務功能。
- 不將需要授權或伺服器計算的功能偽裝成可離線使用。
- 不進行與課程入口及載入韌性無關的全面視覺改版。

## 採用方案

採用「前端韌性載入＋後端冷啟動拆分」。僅增加前端逾時雖能避免永久載入，卻無法改善登入與互動功能的等待時間；只更換主機則無法解決請求無逾時與初始化耦合。因此兩端必須一起調整。

## 使用者體驗

### 三階段載入

1. 0–3 秒：顯示「正在連線課程平台」，保留穩定版面並說明正在確認登入狀態。
2. 3–8 秒：顯示「正在喚醒雲端服務」，告知教材可先瀏覽，並提供「先瀏覽課程」與「重新連線」。
3. 超過 8 秒：顯示「雲端暫時無法連線」，提供「進入離線課程」、「再次嘗試」及「登出並返回登入」。

載入介面不得只顯示動畫或技術錯誤。每個狀態都必須說明正在發生什麼、學生可以做什麼，以及資料是否安全。

### 離線課程模式

可用功能：

- 課程首頁與 18 週課程導覽。
- 前端內建的課程標題、概念卡與視覺化元件。
- 已打包進前端的靜態說明。

暫停功能：

- AI 助教與聊天串流。
- 測驗取得、提交與成績保存。
- 後端提供的教材下載。
- 儀表板、管理設定與使用者管理。

暫停功能保留原本位置，但改為局部狀態卡，說明需等待雲端；不得阻塞整個課程頁。雲端恢復後，頁面原地切回完整模式，不強制重新整理或遺失目前週次。

### 視覺原則

- 連線中使用藍色，喚醒中使用琥珀色，無法連線使用紅色，但狀態不能只靠顏色辨識。
- 使用真實狀態文字與進度階段，不顯示無法證實的百分比。
- 按鈕需有鍵盤焦點、可讀標籤與足夠觸控尺寸。
- 手機版採單欄配置，主要操作維持在首個視窗內。

## 前端架構

### API 請求層

`fetchAPI` 擴充為共用請求介面，負責：

- 預設逾時與呼叫端可覆寫的逾時設定。
- 使用 `AbortController`，並合併呼叫端取消訊號。
- 將失敗分類為 `unauthorized`、`timeout`、`offline`、`server` 與 `unknown`。
- 保留 HTTP 狀態與可供 UI 顯示的安全訊息。

身份驗證啟動請求使用 8 秒逾時。頁面卸載、登出或新請求取代舊請求時，必須取消舊請求。

### 身份與服務狀態

身份狀態與雲端狀態分開建模：

- 身份：`checking | authenticated | anonymous | unverified`。
- 雲端：`connecting | waking | ready | degraded | unavailable`。

`unverified` 表示本機有 token，但因 timeout、斷線或伺服器錯誤而無法驗證。它允許離線課程，不等同登入成功，也不得開放受保護管理功能。

只有後端明確回覆 `401` 或 token 本地判定為不可用時才移除 token。timeout、斷線與 `5xx` 必須保留 token，避免服務故障造成不必要登出。

### 路由保護

路由依功能分級：

- 公開／離線可用：登入、首頁、週次課程頁。
- 已驗證才可用：學習儀表板與需要保存的學生操作。
- 角色授權才可用：管理設定、使用者管理、題庫管理。

離線可用路由不等待身份驗證完成。已驗證及管理路由在 `unverified` 時顯示服務狀態與重試操作，不渲染敏感內容。

### 重試策略

背景健康檢查採約 2、4、8、15 秒的漸進間隔，再維持最高 15 秒；每次加入少量隨機差異，避免同班學生同時重試。使用者手動重試可立即執行一次，但同一時間僅允許一個請求。離開頁面、登出或連線恢復時停止排程。

## 後端架構

### 啟動分層

必要啟動僅包含：

- FastAPI 應用與路由建立。
- 最小資料庫連線與必要 schema 準備。
- 健康狀態容器初始化。

教材掃描、RAG 自動索引及網路知識更新移至受控背景任務。背景任務失敗不得使程序退出，也不得讓 `/health` 無法回應。若 SQLite schema 初始化本身超過合理時間，健康狀態應停留在 `starting` 並記錄該階段，而不是靜默等待。

### 健康檢查

`/health` 回傳：

- `status`: `starting | ready | degraded`。
- `database`: `connected | error`。
- `rag`: `pending | indexing | ready | error`。
- `uptime_seconds`。

HTTP 規則：基礎服務能接受請求時回傳 `200`，即使 RAG 尚未完成；資料庫等核心依賴不可用時回傳 `503`。健康檢查不得等待 RAG、外部 API 或長時間鎖定的工作。

### 可觀測性

記錄下列結構化事件：

- 程序啟動至路由可服務的時間。
- 資料庫、RAG 表格、教材索引及背景排程各階段耗時與結果。
- 每個請求的路徑、狀態碼與耗時。
- 背景工作失敗類型及下一次重試時間。

不得記錄密碼、JWT、Authorization header、聊天完整內容或其他敏感資料。

## 錯誤處理

- `401`：清除 token，導向登入並顯示登入已失效。
- timeout：保留 token，切換為喚醒或離線課程狀態。
- 瀏覽器 offline：停止立即網路重試；收到 `online` 事件後恢復。
- `5xx`：保留 token，顯示服務暫時異常並允許離線瀏覽。
- 單一課程元件載入失敗：只顯示該區塊的重試卡，由 Error Boundary 隔離。
- RAG 背景初始化失敗：AI 功能顯示降級狀態，其他 API 保持可用。

## 測試策略

實作採測試先行，每項行為先確認測試因缺少功能而失敗。

### 前端單元與整合測試

- `/api/auth/me` 永不完成時，8 秒後可進入離線課程。
- 3 秒與 8 秒時分別顯示正確狀態與操作。
- `401` 清除 token；timeout、offline 與 `5xx` 保留 token。
- 重試間隔正確，且取消、卸載及恢復後不殘留 timer 或請求。
- 離線課程可開啟 18 週頁面，受保護功能不可使用。
- 雲端恢復後原地切換為完整功能並保留目前路由。
- 視覺化元件失敗只影響自身區塊。

### 後端測試

- 模擬慢速 RAG 索引時，`/health` 仍快速回應 `starting` 或 `degraded`。
- RAG 失敗不終止應用，核心 API 仍可服務。
- 資料庫失敗時 `/health` 回覆 `503` 與安全錯誤狀態。
- 啟動階段狀態依序更新，背景工作不重複啟動。

### 驗收與效能界線

- 前端永遠不會無限顯示 `Loading...`。
- 雲端無回應時，使用者在 8 秒內取得可操作的離線課程入口。
- 後端程序開始監聽後，`/health` 在一般狀況下 1 秒內回應；此界線不包含供應商尚未啟動容器的時間。
- 冷啟動原因可從日誌分辨為供應商喚醒、核心初始化或背景索引。
- 執行完整前後端測試、型別檢查、lint 與正式建置，並以桌面及手機尺寸手動驗證主要流程。

## 推出方式

1. 先加入前端請求分類、身份狀態與失敗測試，避免永久載入。
2. 加入離線課程路由與三階段 UI。
3. 拆分後端啟動流程並擴充健康檢查。
4. 加入啟動量測與背景任務狀態。
5. 在本機模擬延遲、無回應、`401`、`5xx` 及 RAG 失敗。
6. 部署後重新量測第一次與第二次 `/health`，確認雲端實際改善幅度。

## 成功定義

學生開啟系統時，即使雲端正在喚醒或暫時故障，也能在明確等待上限內進入課程內容；雲端恢復後互動功能自動回來。維運者能從健康狀態與日誌指出延遲發生在哪一階段，而不再以永久 `Loading...` 隱藏問題。
