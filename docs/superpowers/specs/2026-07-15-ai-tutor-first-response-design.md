# AI 助教快速首答與完整精煉設計

日期：2026-07-15

## 問題

目前每次提問都建立新的 WebSocket。後端在送出第一個 chunk 前，會同步完成 RAG 檢索、提示組裝及完整 42 層 NLP pipeline；本地 provider 產生整份答案後才切句模擬串流。因此使用者長時間只看到載入提示，實際上沒有首字串流。

## 目標

- 本地 AI 在收到問題後 1 秒內提供 2–4 句可用的快速回答。
- 完整 RAG 與 42 層 NLP pipeline 保留，負責最終精煉答案。
- 前端清楚區分尚未完成教材驗證的快速回答與正式答案。
- 分別量測連線、快速檢索、首答與完整答案耗時。

## 非目標

- 本次不建立永久 WebSocket 或在面板開啟時預熱模型。
- 不移除或縮減現有 42 層 NLP pipeline。
- 不更換 LLM provider、模型或雲端主機。
- 不改寫整套 RAG 排序策略。

## 採用方案

採用雙階段串流：快速路徑先回覆核心答案，完整路徑隨後以現有 RAG 與 NLP pipeline 產生精煉答案。前端使用同一則 AI 訊息呈現兩個階段，精煉內容取代草稿，避免重複訊息。

## WebSocket 事件協定

伺服器依序送出：

1. `{ "type": "status", "stage": "analyzing" }`
2. `{ "type": "draft", "content": "...", "elapsed_ms": 123 }`
3. `{ "type": "status", "stage": "verifying" }`
4. 一或多個 `{ "type": "refinement", "content": "..." }`
5. `{ "type": "done", "elapsed_ms": 840, "draft_ms": 123 }`

失敗事件使用 `{ "type": "error", "stage": "draft|refinement", "content": "安全訊息" }`。既有 `chunk` 事件在遷移期間仍由前端視為 `refinement`，避免其他 provider 或舊後端造成空白。

## 快速路徑

- 輸入僅使用最新問題、目前週次與課程主題。
- RAG 最多取 2 個目前週次教材片段，內容上限 1,200 字元。
- 不載入完整 42 層 pipeline，不執行情緒、學習風格、跨週延伸或完整引用組裝。
- 本地 provider 使用規則化快速回答器，從問題意圖、課程關鍵詞與教材片段形成 2–4 句核心回答。
- 外部 provider 不額外發出第二次付費請求；若無安全且快速的本地草稿，僅送出狀態事件並直接進入正式 provider 串流。
- 醫療、法律、財務、自傷及其他高風險或明顯超出課程範圍的問題不得生成推測性草稿，改為簡短範圍提示並等待完整答案。

## 完整路徑

- 沿用 `AITutor._build_system` 的完整 RAG 上下文。
- 本地 provider 執行既有 42 層 NLP pipeline。
- Anthropic 與 OpenAI provider 沿用原生串流。
- 完整答案開始前送出 `verifying`，前端保留草稿直到第一個 refinement 到達。

## 前端體驗

- 送出後立即顯示「正在理解問題」。
- 收到草稿後顯示內容與「快速回答」標籤。
- 收到 `verifying` 後顯示「正在核對教材」，但不隱藏草稿。
- 第一個 refinement 到達時清空草稿並開始組裝正式答案。
- `done` 後顯示「已完成教材驗證」。
- 草稿失敗時不顯示錯誤泡泡，只繼續等待正式答案。
- 正式答案失敗時保留草稿，標記「尚未完成教材驗證」，並提供重試。

## 後端邊界

- 新增獨立快速回答單元，輸入與輸出均為明確型別，不依賴 WebSocket。
- 路由負責事件順序與計時；provider 只負責內容生成。
- 快速路徑不得修改對話記憶、學習紀錄或 RAG 資料。
- 同一連線斷開時取消尚未完成的工作。

## 可觀測性

記錄：provider、week、連線至收題時間、draft_ms、first_refinement_ms、total_ms、draft 是否成功。不得記錄問題全文、答案全文、JWT 或學生識別資料。

## 錯誤與降級

- 快速檢索無結果：使用課程主題形成保守草稿，或略過草稿。
- 快速路徑超過 800ms：取消草稿並直接等待完整串流。
- 完整路徑失敗：保留草稿並標示未驗證。
- WebSocket 中斷：前端停止載入狀態並提供重試；後端取消任務。
- 舊事件格式：前端把 `chunk` 當作 refinement。

## 測試

- 快速回答器在無需完整 pipeline 的情況下產生 2–4 句課程內答案。
- 高風險及課程外問題不產生推測性草稿。
- WebSocket 事件順序為 analyzing、draft、verifying、refinement、done。
- 草稿超時或失敗仍會收到完整答案。
- 前端草稿與精煉共用一則 AI 訊息，第一個 refinement 會取代草稿。
- 完整失敗時保留草稿及未驗證標示。
- 舊 `chunk` 事件仍可顯示。
- 現有前後端測試、lint、正式建置全部通過。

## 成功標準

- 本地測試環境的 draft 首答 p95 小於 1,000ms，快速路徑 timeout 為 800ms。
- 使用者送出後立即看到處理階段，且草稿到達前不出現無狀態空白。
- 最終答案仍執行既有完整 pipeline。
- 可從日誌分辨延遲位於快速檢索、完整 NLP 或外部 provider。
