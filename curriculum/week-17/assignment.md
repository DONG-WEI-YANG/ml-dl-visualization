## 作業一：文字嵌入探索與視覺化（20%）

### 任務描述
使用嵌入模型將一組文本轉換為向量，並進行視覺化分析。

### 要求
1. 準備至少 **30 句**涵蓋 **3 個不同主題** 的中文或英文句子（每主題至少 10 句）
   - 範例主題：機器學習、烹飪食譜、體育運動
2. 使用嵌入模型（`sentence-transformers/all-MiniLM-L6-v2` 或 OpenAI Embeddings）生成嵌入向量
3. 使用 **t-SNE 或 UMAP** 將高維嵌入降維至 2D
4. 繪製散佈圖，依主題著色，並分析：
   - 同主題的句子是否形成聚類？
   - 不同主題之間的距離是否符合直覺？
   - 是否有跨主題的語義重疊？
5. 計算同主題與跨主題的**平均餘弦相似度**，以數據佐證視覺化觀察

### 加分項（Bonus +5%）
- 比較兩種不同嵌入模型的結果差異

## 作業二：語義搜尋系統（20%）

### 任務描述
建構一個簡易的語義搜尋系統。

### 要求
1. 準備一個小型文件庫（至少 **20 段**文字，可來自維基百科、教科書或自行撰寫）
2. 將文件庫嵌入為向量並建立索引（使用 Chroma、Faiss 或 NumPy）
3. 實作語義搜尋功能：
   - 輸入查詢句子
   - 返回 Top-5 最相關段落及其相似度分數
4. 設計至少 **5 個測試查詢**，包含：
   - 直接匹配的查詢（文件庫中有高度相關內容）
   - 同義詞/改述查詢（用不同詞彙表達相同概念）
   - 無關查詢（文件庫中無相關內容）
5. 展示搜尋結果並分析品質

### 加分項（Bonus +5%）
- 實作混合搜尋（結合關鍵字搜尋 + 語義搜尋）

## 作業三：簡易 RAG 系統（25%）

### 任務描述
建構一個完整的 RAG 問答系統。

### 要求
1. 選擇一個主題領域並準備知識文件（建議 3-5 頁的文字內容）
   - 可使用課程教材、維基百科文章、技術文件等
2. 實作完整的 RAG 流程：
   - **文件切割**：使用適當的 chunking 策略，說明選擇理由
   - **嵌入與儲存**：將 chunks 嵌入並存入向量資料庫
   - **檢索**：根據使用者問題檢索相關段落
   - **生成**：使用 LLM API 根據檢索結果生成回答
3. 設計一個有效的 **System Prompt**，包含角色定義、行為準則、輸出格式
4. 測試至少 **5 個問題**，展示系統的回答品質
5. 撰寫分析報告：
   - 切割策略的選擇與參數設定
   - 檢索結果的品質評估
   - 哪些問題回答得好？哪些不好？為什麼？

### 注意事項
- 如果沒有 LLM API Key，可使用模擬的 LLM 回應（將檢索到的段落整理為回答）
- 重點在於理解 RAG 流程，而非依賴特定 API

## 作業四：Prompt Engineering 實驗（20%）

### 任務描述
系統性地比較不同提示策略對 LLM 輸出品質的影響。

### 要求
1. 選擇以下任一任務類型：
   - (a) 文本分類（情感分析或主題分類）
   - (b) 問答（需要推理的問題）
   - (c) 文本摘要
   - (d) 程式碼生成

2. 準備至少 **10 個測試案例**

3. 對每個案例，分別使用以下提示策略：
   - **Zero-shot**：直接描述任務
   - **Few-shot (3-shot)**：提供 3 個範例
   - **Zero-shot CoT**：加入「讓我們一步一步思考」
   - **Few-shot CoT**：範例中包含推理過程

4. 記錄每種策略的輸出，從以下維度進行比較：
   - 正確性 (Correctness)
   - 完整性 (Completeness)
   - 一致性 (Consistency)

5. 撰寫實驗報告：
   - 哪種提示策略效果最好？在什麼情況下？
   - 範例的品質和數量如何影響結果？
   - 你發現了哪些有趣的現象？

### 注意事項
- 如果沒有 LLM API Key，可使用免費的 LLM 服務（如 Hugging Face Inference API, Google Gemini 免費方案）或在報告中描述預期結果
