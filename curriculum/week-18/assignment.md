## 一、專題報告格式 Project Report Format

### 1.1 繳交項目清單 Submission Checklist

請確認你的 GitHub Repository 包含以下所有項目：

```
your-project/
├── README.md                    # 專案說明文件（必繳）
├── requirements.txt             # 套件依賴清單（必繳）
├── notebooks/
│   └── final_project.ipynb      # 主要分析 Notebook（必繳）
├── src/                         # 原始碼目錄（必繳）
│   ├── data_processing.py       # 資料前處理
│   ├── model.py                 # 模型定義與訓練
│   └── utils.py                 # 工具函式
├── data/
│   ├── README.md                # 資料說明
│   └── sample.csv               # 範例資料（若資料太大請提供子集）
├── results/
│   ├── figures/                 # 圖表輸出
│   └── metrics.json             # 評估指標結果
├── slides/
│   └── presentation.pdf         # 簡報檔案（必繳）
├── docs/
│   ├── ethics_checklist.md      # 倫理檢核清單（必繳）
│   └── reflection.md            # 學習反思（必繳）
├── Dockerfile                   # Docker 部署文件（選繳，加分）
├── .gitignore                   # Git 忽略規則（必繳）
└── LICENSE                      # 授權聲明（選繳）
```

### 1.2 README.md 必要內容 Required README Content

你的 README.md **至少**需包含以下段落：

1. **專案標題與一句話描述**
2. **問題描述 Problem Description**：為什麼要做這個專案？解決什麼問題？
3. **資料集 Dataset**：來源、大小、特徵描述、授權
4. **安裝與執行 Installation & Usage**：
   - 環境需求（Python 版本、CUDA 版本）
   - 安裝步驟（pip install -r requirements.txt）
   - 執行指令
5. **專案結構 Project Structure**：目錄樹說明
6. **方法 Methodology**：使用的模型與方法簡述
7. **實驗結果 Results**：關鍵評估指標表格
8. **倫理聲明 Ethics Statement**：簡要說明
9. **貢獻者 Contributors**：姓名與貢獻分工

### 1.3 Notebook 撰寫規範 Notebook Guidelines

- Notebook 必須能 **Restart & Run All** 成功
- 每個分析步驟前需有 **Markdown 說明**，解釋做什麼、為什麼做
- 關鍵程式碼需有 **行內註解**
- 圖表需有 **標題、軸標籤、圖例**
- 輸出結果需有 **文字解讀**
- 避免留下除錯用的 print 或無用的 cell

### 1.4 倫理檢核清單 Ethics Checklist

請在 `docs/ethics_checklist.md` 中回答以下問題（每題至少 2-3 句）：

**資料面 Data:**
1. 你的資料來源合法嗎？是否涉及個人隱私？
2. 資料中是否存在已知的偏誤？你如何處理？
3. 資料是否已適當去識別化 (De-identified)？

**模型面 Model:**
4. 你的模型對不同群體（如性別、年齡、種族）的表現是否一致？
5. 你能解釋模型為什麼做出特定預測嗎？使用了什麼方法？
6. 模型的錯誤預測可能造成什麼負面影響？

**應用面 Application:**
7. 你的模型預期的使用場景是什麼？有什麼不適合的場景？
8. 模型是否可能被誤用？你有什麼建議的防範措施？
9. 若要部署此模型，你認為需要什麼樣的人類監督機制？

### 1.5 學習反思 Learning Reflection

請在 `docs/reflection.md` 中回答以下問題（每題至少 3-5 句）：

1. **學習收穫**：這學期你學到最重要的 3 件事是什麼？
2. **專題挑戰**：在做專題過程中，你遇到最大的挑戰是什麼？你如何克服的？
3. **技術成長**：比較學期初和學期末，你的技術能力有什麼變化？
4. **未來方向**：這門課對你未來的學習/職涯規劃有什麼影響？
5. **自我評價**：如果你給自己這學期的學習打分（1-10），你會打幾分？為什麼？

## 二、程式碼繳交規範 Code Submission Guidelines

### 2.1 程式碼品質要求 Code Quality Requirements

| 項目 | 要求 |
|------|------|
| 可執行性 | 按照 README 步驟可成功執行 |
| 隨機種子 | 所有隨機操作需固定 seed（建議 42） |
| 程式風格 | 遵循 PEP 8 規範 |
| 函式設計 | 重複邏輯封裝為函式，避免 copy-paste |
| 命名規範 | 變數名有意義（避免 a, b, x1 等） |
| 註解 | 關鍵函式有 docstring，複雜邏輯有行內註解 |
| 版本控制 | 有意義的 commit messages |

### 2.2 requirements.txt 規範

```
# 請使用精確版本號
numpy==1.26.4
pandas==2.2.1
scikit-learn==1.4.1
matplotlib==3.8.3
seaborn==0.13.2
# ... 其他依賴
```

- 使用 `pip freeze` 或 `pipreqs` 生成
- 確保版本號固定（使用 `==`）
- 不要把整個環境 freeze 上去（只包含專案需要的套件）

### 2.3 .gitignore 規範

至少應忽略以下內容：
- 大型資料檔案（>50MB）
- 模型權重檔案（>50MB）
- 環境目錄（venv/, .env）
- 系統檔案（.DS_Store, Thumbs.db）
- Jupyter checkpoints
- IDE 設定檔

### 2.4 版本控制 Git Best Practices

- 不要在最後一天把所有東西一次 commit
- 每個有意義的步驟做一次 commit
- Commit message 使用英文，簡潔明瞭
- 範例：`Add EDA notebook with correlation analysis`
- 不要 commit 機密資訊（API keys, passwords）

## 三、展示時間與格式 Presentation Format

### 3.1 展示規格 Presentation Specifications

| 項目 | 規格 |
|------|------|
| 展示時間 | 每組 15 分鐘（12 分鐘展示 + 3 分鐘 Q&A） |
| 簡報格式 | PDF 或 PPTX，16:9 比例 |
| 投影片數量 | 建議 12-15 張 |
| 字體大小 | 標題 >= 28pt，內文 >= 18pt |
| 團隊規模 | 1-3 人 |

### 3.2 展示內容要求 Required Content

你的展示**必須**涵蓋以下內容：

1. **封面頁**：專題名稱、成員姓名、日期
2. **問題定義與動機**：為什麼做這個題目
3. **資料介紹**：資料來源、大小、關鍵特徵
4. **EDA 重點發現**：1-2 張關鍵圖表
5. **方法說明**：模型選擇與理由
6. **實驗結果**：效能比較表、關鍵圖表
7. **模型解釋**：SHAP 或其他 XAI 結果
8. **Demo（建議）**：現場展示互動功能
9. **倫理考量**：簡要說明
10. **結論與限制**：主要發現、限制、未來方向

### 3.3 展示注意事項 Presentation Tips

- **所有成員都要發言**：分配各段落的講者
- **練習時間**：至少完整排練 2 次
- **準備備案**：Demo 可能失敗，準備截圖/影片備案
- **提前到場**：展示前 15 分鐘到教室測試設備
- **面向觀眾**：不要背對觀眾看螢幕
- **避免唸稿**：用關鍵字提醒自己，不要逐字唸投影片

### 3.4 Q&A 準備 Q&A Preparation

預先準備以下問題的回答：
1. 你為什麼選擇這個模型？
2. 你如何處理資料中的偏誤？
3. 模型的限制是什麼？
4. 如果要部署，你會怎麼做？
5. 下一步你會做什麼改進？

## 四、同儕互評表 Peer Review Form

### 4.1 互評時間 Peer Review Timing

- 在所有組別展示結束後統一填寫
- 填寫時間：約 10 分鐘
- 每位同學需對**除自己以外的所有組別**各填一份

### 4.2 互評內容 Peer Review Content

對每一組，請評分以下四個面向（1-5 分）：

| 面向 | 1 分 | 3 分 | 5 分 |
|------|------|------|------|
| 展示品質 Presentation | 結構不清、表達混亂 | 結構完整、表達尚可 | 結構精彩、表達流暢 |
| 技術內容 Technical | 方法簡單、分析不足 | 方法合理、分析完整 | 方法深入、見解獨到 |
| 創意印象 Creativity | 題目常見、無新意 | 有一定的創新 | 題目獨特、方法新穎 |
| 整體印象 Overall | 不推薦 | 一般 | 強烈推薦 |

### 4.3 文字回饋 Written Feedback

對每一組，請回答：
1. 最讓你印象深刻的部分是什麼？（至少一句）
2. 你認為可以改進的地方是什麼？（至少一句）

### 4.4 互評注意事項 Peer Review Guidelines

- 請根據展示內容客觀評分，不要受私人關係影響
- 文字回饋請具體且建設性
- 不要所有面向都給相同分數，請仔細區分各面向
- 互評分數將計入該組的期末專題成績（佔 10%）
- 互評為匿名，但教師可查看以確保品質

## 常見問答 FAQ

### Q: 可以使用 Kaggle 上的 Kernel 嗎？
A: 可以參考，但不可以直接複製。你的 Notebook 需展現自己的分析思路，並在 README 中引用參考來源。

### Q: 資料集有大小限制嗎？
A: GitHub 建議單檔 < 50MB。若資料較大，請提供下載腳本或在 README 中說明取得方式，並在 `data/` 中放小型範例資料。

### Q: 我可以使用預訓練模型嗎？
A: 可以，但需清楚說明使用了哪個預訓練模型、為什麼選擇它、以及你做了哪些調整（如 fine-tuning）。

### Q: 團隊中每個人的貢獻如何評估？
A: README 中需說明每位成員的分工。展示時所有成員都需發言。若教師認為分工明顯不均，保留個別調整成績的權利。

### Q: 展示可以用中文嗎？
A: 可以。簡報以中文為主，專有名詞附英文。程式碼註解與變數名建議用英文。

### Q: 如果展示日無法出席怎麼辦？
A: 請提前告知教師。可安排補展或接受錄影展示，但 Demo 互動分數會受影響。
